1. Introdução

O "Delivery" nasceu da necessidade de conectar restaurantes locais, entregadores autônomos e clientes finais em uma plataforma única. Os resultados alcançados no protótipo indicam redução de 40% no tempo médio de entrega e suporte a reembolsos automatizados via análise de foto do pedido.

2. Visão geral desse documento

Este documento organiza os requisitos em visões: pedidos, entregas, reembolso. Setores envolvidos: operações, TI, financeiro. A abordagem segue FURPS+ adaptado.

3. Convenções, termos e abreviações

OPE Operador de atendimento · ENT Entregador · CLI Cliente · REEMB Reembolso · SLA Acordo nível serviço

4. Identificações dos requisitos

Requisitos funcionais: pedidos e pagamentos [RF]; Requisitos Não funcionais: desempenho, segurança e alta Disponibilidade [NF];

5. Prioridade dos requisitos

Alta (Essencial), Média (Importante), Baixa (Desejável). Ex: reembolso → alta; relatório mensal → média.

6. Referências

Atas de reunião 05/2025; benchmark iFood; documentação API de pagamento.

7. Descrição geral do sistema

"Delivery" é um sistema web responsivo. Gerencia pedidos, atribui entregadores, processa pagamentos e reembolsos. Tecnologias previstas: React, Node.js.

8. Descrição dos usuários

a) Usuário A – Cliente Solicita pedidos, acompanha entrega, pede reembolso. Cerca de 1000 usuários ativos. ㅤㅤㅤb) Usuário B – Entregador Aceita corridas, visualiza rota, confirma retirada e entrega. Meta de 100 entregadores.ㅤ c) Usuário C – Operador gerencia reembolsos, suporte, análise de disputas.ㅤㅤㅤㅤㅤㅤㅤㅤㅤㅤㅤㅤㅤㅤㅤㅤd) Usuário D – Restaurante Confirma pedidos, atualiza cardápio.

9. Requisitos funcionais

[RF001] Funcionalidade de pedido e pagamento 01:
Cliente seleciona itens, realiz a checkout, paga via cartão ou PIX. Destinado a ator CLIENTE. Prioridade: alta.
[RF002] Rastreamento de entregas em tempo real 02:
Entregador (ator ENT) compartilha localização; cliente visualiza no mapa. Atualização a cada 10s.. Prioridade: média.
[RF003] solicitação de reembolso 03:
Cliente abre solicitação de reembolso anexando foto do problema. Operador analisa e aprova estorno. Categoria financeiro..
[RF004] Gerenciamento de entregadores 04:
Administrador cadastra, bloqueia ou altera modo de disponibilidade..

10. Requisitos não funcionais

[NF001] Requisito não funcional 01:
Autenticação para operadores. (segurança) – prioridade alta.
[NF002] Requisito não funcional 02:
Dados de pagamento - prioridade alta.
[NF003] Requisito não funcional 03:
Suporte a 1000 pedidos simultâneos (confiabilidade). Prioridade média.
