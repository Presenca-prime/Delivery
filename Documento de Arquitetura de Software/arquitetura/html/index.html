1. INTRODUÇÃO

1.1 Finalidade

O sistema "Presença prime - delivery" tem como finalidade conectar clientes a restaurantes parceiros e entregadores autônomos, permitindo pedidos de refeições com rastreamento em tempo real. O projeto foi idealizado para otimizar a experiência de delivery, oferecendo um aplicativo intuitivo para o cliente, um painel de controle para a administração e um aplicativo dedicado para entregadores.

1.2 Escopo

Esta documentação descreve a arquitetura de software do sistema de delivery, abrangendo os subsistemas de cliente (mobile app), entregador (mobile app) e administração (web dashboard). O documento apresenta as decisões arquiteturais, tecnologias utilizadas, fluxos de informação e a interação entre os componentes.

1.3 Definições, Acrónimos e Abreviações

• API: Interface de Programação de Aplicações
• BPMN: Business Process Model and Notation
• Cliente: Usuário que solicita o delivery
• Entregador: Profissional autônomo que realiza a entrega
• Admin: Gestor responsável pela administração do sistema (restaurantes, entregadores, finanças)
• Backbone: Funcionalidades essenciais para o funcionamento do sistema

2. REPRESENTAÇÃO ARQUITETURAL

A arquitetura é baseada em microsserviços, com separação clara entre os domínios: cliente, entregador e administração. A comunicação é feita via APIs REST e mensageria assíncrona (para atualizações de status). Esta visão é destinada a desenvolvedores, arquitetos de software e gestores de TI, demonstrando a interação entre front-end (aplicativos) e back-end (serviços).

3. REQUISITOS E RESTRIÇÕES ARQUITETURAIS

• Linguagem: Java (Spring Boot) para microsserviços; React Native para apps mobile; React para web admin.
• Plataforma: AWS (Amazon Web Services) com containers Docker e orquestração Kubernetes.
• Segurança: Autenticação via JWT (JSON Web Tokens), criptografia de dados sensíveis, HTTPS em todas as comunicações.
• Restrições: Latência máxima de 2s para atualização de status de entrega; disponibilidade de 99.9%; escalabilidade para 2000 pedidos simultâneos.

📋 Casos de Uso em Detalhe:

UC01 - Realizar Pedido (Cliente)

O cliente acessa o app, navega pelos restaurantes, seleciona itens e confirma o pedido. O Serviço de Pedidos cria a solicitação, aciona o Serviço de Pagamento e notifica os entregadores disponíveis via mensageria.

UC02 - Gerenciar Entregas (Entregador)

O entregador visualiza corridas disponíveis em tempo real, aceita uma entrega e atualiza o status (a caminho, entregue). O Serviço de Rastreio compartilha a localização com o cliente.

UC03 - Administrar Plataforma (Admin)

O administrador acessa o dashboard web para gerenciar restaurantes parceiros, visualizar métricas de desempenho, gerar relatórios financeiros e resolver disputas.

BPMN - Fluxo Completo

O processo se inicia com o pedido do cliente, passa pela validação de pagamento, busca de entregador, preparo do pedido no restaurante, coleta e entrega, finalizando com a confirmação e avaliação.

Fluxo Arquitetural:
1. Cliente realiza pedido → API Gateway → Serviço de Pedidos
2. Serviço de Pedidos → Serviço de Pagamento (validação)
3. Pagamento aprovado → Mensageria notifica entregadores próximos
4. Entregador aceita → Serviço de Rastreio ativa tracking
5. Restaurante prepara → Status atualizado para cliente
6. Entrega concluída → Serviço de Relatórios atualiza métricas

5. VISÃO LÓGICA

Principais classes do sistema:
• Cliente (id, nome, endereço, históricoPedidos)
• Entregador (id, nome, veículo, status, localização)
• Pedido (id, itens, valorTotal, status, clienteId, entregadorId)
• Restaurante (id, nome, cardápio, endereço)
• Pagamento (id, pedidoId, método, status)
• Relacionamentos: Pedido pertence a um Cliente e pode ser associado a um Entregador; Restaurante fornece os itens do Pedido.
